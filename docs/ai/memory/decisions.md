- 2026-03-30 — Decision: **Vitest over Jest** — Rationale: ESM-native, fast startup, good TypeScript integration without extra transform config. jsdom for DOM environment since plugin uses Obsidian UI APIs
- 2026-03-30 — Decision: **Obsidian API mock via Vitest alias** — Rationale: Obsidian can only be used after consultation; mock allows unit testing without Obsidian runtime. Alias in vitest.config.ts redirects `obsidian` imports to `test/__mocks__/obsidian.ts`
- 2026-03-30 — Decision: **semantic-release (Dynbedded pattern)** — Rationale: Reference repo obsidian-dynbedded uses same approach. Conventional Commits drive automatic versioning, changelog, and GitHub Release with plugin assets. No npm publish needed
- 2026-03-30 — Decision: **Build output to `build/` for releases** — Rationale: Separates dev output (root `main.js` for hot-reload) from release artifacts (build/ dir with all assets for GitHub Release upload)
- 2026-04-20 — Decision: **`kado-open-notes` two-layer gate: feature-flag AND path-ACL, no inheritance** — Rationale: Spec 006. Per-key `allowActiveNote`/`allowOtherNotes` AND-combine with global (no inheritance). Feature-gate denial returns FORBIDDEN with explicit reason; path-ACL denial is silent (no note-existence leak). Default OFF everywhere. See `docs/XDD/specs/006-open-notes-tool/solution.md` for ADRs 1–6
- 2026-05-09 — Decision: **`resolveScope` picks the most specific matching pattern, not the first-declared** — Rationale: First-match-wins made vault-wide patterns like `**` (default-deny / read-only fallback) silently shadow narrower exceptions like `X/900 Support/**`, even when the narrow rule was declared explicitly. Specificity score = count of literal (non-`*`/`?`) characters in the pattern; declaration order is the deterministic tie-breaker. Applies in both whitelist and blacklist modes (global security AND per-key scope). Implementation: `src/core/gates/scope-resolver.ts`
- 2026-05-20 — Decision: **`kado-write operation=frontmatter` default mode is deep-merge; arrays REPLACE (no concat)** — Rationale: Tomo F-47 needs frontmatter-only state transitions (`tomo.state: pending → approved`) without a read+regex+write round-trip. Pre-0.9.6 behaviour was shallow `Object.assign`, which clobbered nested objects whole — wrong for `tomo: {state, doc_type, run_id}` style structured frontmatter. New default `mode=merge` recurses into nested objects (preserves untouched sibling keys); explicit `mode=replace` clears the existing block. Arrays always replace because lifecycle tags are replacement-style (`#captured → #active`); concat-merge would be a silently wrong default and clients that need set-arithmetic can do it client-side (YAGNI). Behaviour change for any pre-0.9.6 caller relying on shallow semantics — switch to `mode=replace` explicitly. Implementation: `src/core/deep-merge.ts`, `src/obsidian/frontmatter-adapter.ts`. Shipped in Kado 0.9.6 (PR #49).
- 2026-05-20 — Decision: **Audit log adds `bodyTouched?: boolean` field, set to `false` only on `kado-write operation=frontmatter`** — Rationale: Tomo asked for a marker distinguishing metadata-only writes from content writes during audit review. Absent on all other entries (no noise added; absence is not a signal). Could be extended to `dataview-inline-field` (body IS touched) and `delete operation=frontmatter` (body NOT touched) later, but kept minimal at first ship. Implementation: `src/core/audit-logger.ts` (AuditEntry shape), `src/mcp/tools.ts` (logAllowed). Shipped in Kado 0.9.6 (PR #49).
- 2026-06-14 — Decision: **`kado-rename` is one tool with two inferred modes; rename → `update`, move → `delete`(source)+`create`(target)** — Rationale: rename/move was the only missing file-level operation (CRUD covered create/read/update/delete, not relocation). Modeled as a single tool whose mode is inferred from the paths (same parent ⇒ rename, different parent ⇒ move) rather than a flag — fewer params, less error surface (YAGNI). The permission split mirrors the trust boundary: an in-folder rename is a form of editing (`update`, checked on both source and target so filename-specific scopes still gate), while a cross-folder move leaves one scope and enters another, so it requires `delete` on the source AND `create` on the target — `update` alone can't move out, `create` alone can't move in. The policy lives in one place (`evaluateRenamePermissions` in `tools.ts`) by composing the **existing** gate chain over synthetic single-path requests, so global-scope/key-scope/path-access/datatype-permission all enforce on every path with zero new gates. Execution uses `app.fileManager.renameFile` (NOT `vault.rename`/`adapter.rename`) — the only API that rewrites inbound `[[wikilinks]]`/markdown links; raw renames break backlinks. **Accepted disclosure boundary:** backlink rewrites touch notes the key may not access; unavoidable (Obsidian updates links vault-wide) and acceptable because it changes references, not content. `expectedModified` guards the source (mirrors delete). Refuses to clobber an existing target (CONFLICT). Implementation: `src/obsidian/rename-adapter.ts`, `src/mcp/tools.ts` (`registerRenameTool`, `evaluateRenamePermissions`), `CoreRenameRequest` (`kind:'rename'`).
- 2026-06-15 — Decision: **kado-rename guards the `fileManager.renameFile` confirmation-dialog hang via conditional registration + timeout, NOT by mutating the vault setting** — Rationale: live testing revealed `app.fileManager.renameFile` hangs forever when Obsidian's "Automatically update internal links" is off (it pops a blocking "update links?" modal an MCP caller can't answer); all 1393 mocked unit tests missed it. Rejected option: temporarily flip `alwaysUpdateLinks` around the rename — changes user state they didn't consent to. Chosen design (per user): (1) **conditional registration** — register `kado-rename` only when `alwaysUpdateLinks` is on OR the opt-in `renameWhenLinkUpdateOff` is on (default off), so by default the tool simply isn't exposed when it would hang; (2) the opt-in is **only shown in settings when auto-update-links is off**, behind a `RenameRiskModal` the user must confirm; (3) a configurable **timeout** (`renameTimeoutMs`, default 60s) makes the handler return a new `TIMEOUT` CoreError instead of hanging; (4) Kado never writes the Obsidian setting. `alwaysUpdateLinks` read via untyped `vault.getConfig` in `src/obsidian/vault-config.ts`; the gate is recomputed in the per-request `registerToolsFn` closure (server.ts `createMcpServer` runs it on every request), so setting changes take effect immediately — no restart. A one-time on-load warning modal (`maybeWarnRenameDisabled`, gated by `renameWarningAcknowledged`) tells the user rename is disabled when auto-update-links is off. Implementation: `src/mcp/tools.ts` (registration gate + `raceWithTimeout`), `src/settings/tabs/GeneralTab.ts`, `src/settings/components/RenameRiskModal.ts`, `KadoConfig.renameWhenLinkUpdateOff`/`renameTimeoutMs`.
- 2026-06-04 — Decision: **`listNotes` outlinks/tags use the source-note disclosure boundary — out-of-scope link targets are returned raw, never resolved** — Rationale: Tomo F-34 (MOC accumulation) needs outlinks + headings + tags per atomic note for topic indexing, sourced from `metadataCache` (no body read). A link target is literal text in the body of an in-scope note the key may already read in full via `operation='note'`, so echoing it as structured data grants no new access — the disclosure boundary is the **source** note, not the target. Filtering out-of-scope targets would be security theater (trivially bypassed by a body read) and would silently degrade Tomo's cross-boundary topic graph. **Invariant:** `listNotes` reads `getFileCache` only for in-scope source notes and returns targets as the raw written string (`cache.links[].link` / `cache.embeds[].link`); it never resolves targets, never reads a target's cache/mtime/existence, and never uses `metadataCache.resolvedLinks`. Same principle for the `tags` field (note's own tags returned in full; `allowedTags` still gates tag-based *selection* via `byTag`/`filter.tags`/`listTags`, not redaction of a readable note's tags). `listNotes` is a `SearchOperation` so it inherits the existing search `note.read` gate (no gate change). Implementation: `src/obsidian/search-adapter.ts` (`listNotes`, `readLinks`, `readHeadings`).
