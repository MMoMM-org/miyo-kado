- 2026-03-30 ‚Äî Decision: **Vitest over Jest** ‚Äî Rationale: ESM-native, fast startup, good TypeScript integration without extra transform config. jsdom for DOM environment since plugin uses Obsidian UI APIs
- 2026-03-30 ‚Äî Decision: **Obsidian API mock via Vitest alias** ‚Äî Rationale: Obsidian can only be used after consultation; mock allows unit testing without Obsidian runtime. Alias in vitest.config.ts redirects `obsidian` imports to `test/__mocks__/obsidian.ts`
- 2026-03-30 ‚Äî Decision: **semantic-release (Dynbedded pattern)** ‚Äî Rationale: Reference repo obsidian-dynbedded uses same approach. Conventional Commits drive automatic versioning, changelog, and GitHub Release with plugin assets. No npm publish needed
- 2026-03-30 ‚Äî Decision: **Build output to `build/` for releases** ‚Äî Rationale: Separates dev output (root `main.js` for hot-reload) from release artifacts (build/ dir with all assets for GitHub Release upload)
- 2026-04-20 ‚Äî Decision: **`kado-open-notes` two-layer gate: feature-flag AND path-ACL, no inheritance** ‚Äî Rationale: Spec 006. Per-key `allowActiveNote`/`allowOtherNotes` AND-combine with global (no inheritance). Feature-gate denial returns FORBIDDEN with explicit reason; path-ACL denial is silent (no note-existence leak). Default OFF everywhere. See `docs/XDD/specs/006-open-notes-tool/solution.md` for ADRs 1‚Äì6
- 2026-05-09 ‚Äî Decision: **`resolveScope` picks the most specific matching pattern, not the first-declared** ‚Äî Rationale: First-match-wins made vault-wide patterns like `**` (default-deny / read-only fallback) silently shadow narrower exceptions like `X/900 Support/**`, even when the narrow rule was declared explicitly. Specificity score = count of literal (non-`*`/`?`) characters in the pattern; declaration order is the deterministic tie-breaker. Applies in both whitelist and blacklist modes (global security AND per-key scope). Implementation: `src/core/gates/scope-resolver.ts`
- 2026-05-20 ‚Äî Decision: **`kado-write operation=frontmatter` default mode is deep-merge; arrays REPLACE (no concat)** ‚Äî Rationale: Tomo F-47 needs frontmatter-only state transitions (`tomo.state: pending ‚Üí approved`) without a read+regex+write round-trip. Pre-0.9.6 behaviour was shallow `Object.assign`, which clobbered nested objects whole ‚Äî wrong for `tomo: {state, doc_type, run_id}` style structured frontmatter. New default `mode=merge` recurses into nested objects (preserves untouched sibling keys); explicit `mode=replace` clears the existing block. Arrays always replace because lifecycle tags are replacement-style (`#captured ‚Üí #active`); concat-merge would be a silently wrong default and clients that need set-arithmetic can do it client-side (YAGNI). Behaviour change for any pre-0.9.6 caller relying on shallow semantics ‚Äî switch to `mode=replace` explicitly. Implementation: `src/core/deep-merge.ts`, `src/obsidian/frontmatter-adapter.ts`. Shipped in Kado 0.9.6 (PR #49).
- 2026-05-20 ‚Äî Decision: **Audit log adds `bodyTouched?: boolean` field, set to `false` only on `kado-write operation=frontmatter`** ‚Äî Rationale: Tomo asked for a marker distinguishing metadata-only writes from content writes during audit review. Absent on all other entries (no noise added; absence is not a signal). Could be extended to `dataview-inline-field` (body IS touched) and `delete operation=frontmatter` (body NOT touched) later, but kept minimal at first ship. Implementation: `src/core/audit-logger.ts` (AuditEntry shape), `src/mcp/tools.ts` (logAllowed). Shipped in Kado 0.9.6 (PR #49).
- 2026-06-04 ‚Äî Decision: **`listNotes` outlinks/tags use the source-note disclosure boundary ‚Äî out-of-scope link targets are returned raw, never resolved** ‚Äî Rationale: Tomo F-34 (MOC accumulation) needs outlinks + headings + tags per atomic note for topic indexing, sourced from `metadataCache` (no body read). A link target is literal text in the body of an in-scope note the key may already read in full via `operation='note'`, so echoing it as structured data grants no new access ‚Äî the disclosure boundary is the **source** note, not the target. Filtering out-of-scope targets would be security theater (trivially bypassed by a body read) and would silently degrade Tomo's cross-boundary topic graph. **Invariant:** `listNotes` reads `getFileCache` only for in-scope source notes and returns targets as the raw written string (`cache.links[].link` / `cache.embeds[].link`); it never resolves targets, never reads a target's cache/mtime/existence, and never uses `metadataCache.resolvedLinks`. Same principle for the `tags` field (note's own tags returned in full; `allowedTags` still gates tag-based *selection* via `byTag`/`filter.tags`/`listTags`, not redaction of a readable note's tags). `listNotes` is a `SearchOperation` so it inherits the existing search `note.read` gate (no gate change). Implementation: `src/obsidian/search-adapter.ts` (`listNotes`, `readLinks`, `readHeadings`).
