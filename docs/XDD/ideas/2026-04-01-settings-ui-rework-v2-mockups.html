4b. Whitelist ↔ Blacklist Flip Behavior

changesv1.md #4 special case — configuration flip, visual indicator change
Step 1 — Starting state: Whitelist mode
Path allowed/** · Note permissions: C R _ X
C R U D
✓ = allowed   empty = not selected   greyed = not available
→ User can Create and Read. Cannot Update (not selected). Delete not available globally.
Step 2 — User flips toggle → Blacklist mode
The listMode changes. The per-path permission booleans do NOT change.
The display rendering maps the same config values differently:
Config value Whitelist display Blacklist display
true checkmark = "allowed" empty = "not blocked"
false empty = "not allowed" X cross = "blocked"
disabled greyed out greyed out
Config: C=T R=T U=F D=—
Display:
C=true → empty (not blocked). R=true → empty (not blocked). U=false → ✕ (blocked). D=disabled → greyed.
→ Same effective permissions as whitelist! User can Create and Read, cannot Update, Delete unavailable.
Step 3 — Why this matters
If the user accidentally flips the toggle, no permissions are lost or changed.
Only the listMode field changes. Permission booleans stay identical.
The display mapping inverts:

Whitelist: true → ✓ checkmark (purple), false → empty
Blacklist: true → empty, false → ✕ cross (red)

The user can then deliberately adjust from the new baseline.
Implementation note: On toggle flip, only listMode changes. Permission booleans are untouched.
UI rendering maps config → display differently per mode:
Whitelist: true → .dot.active (✓)  |  Blacklist: false → .dot.active-block (✕)

Data Model Impact Summary

What changes in the type system
// OLD: Multiple GlobalAreas, permissions per area globalAreas: GlobalArea[] // each with { id, label, pathPatterns, permissions, listMode, tags } // NEW: Single security scope, permissions per path security: { listMode: 'whitelist' | 'blacklist'; paths: Array<{ path: string; permissions: DataTypePermissions }>; tags: string[]; // read-only filters, stored without '#' } // OLD: Key references area by ID areas: Array<{ areaId: string; permissions: DataTypePermissions; tags: string[] }> // NEW: Key has own paths/tags subset with constrained permissions // listMode is INDEPENDENT per key (not inherited from global) listMode: 'whitelist' | 'blacklist'; // key's own toggle paths: Array<{ path: string; permissions: DataTypePermissions }>; tags: string[]; // subset of global tags
No migration needed — update test vault config after code is done.
Audit log: timestamp field changes from number (epoch ms) to string (ISO 8601 with timezone).
Enforcement impact: Global and Key each have independent listMode — 4 combinations possible (see below).

Enforcement Logic — Independent ListMode Combinations

Global and Key each have their own whitelist/blacklist toggle — 4 combinations
The enforcement engine must handle all 4 combinations. The key principle:
Global defines the ceiling (what's possible). Key defines the subset (what this key gets).
Both layers apply their own listMode interpretation independently, then the result is intersected.
Combo 1: Global=Whitelist, Key=Whitelist (most common)
Global: "Only allowed/** with Note:CRUD is accessible"
Key: "Only allowed/** with Note:CR is accessible for this key"
Result: Key can Create and Read notes in allowed/**. Straightforward intersection.
Combo 2: Global=Whitelist, Key=Blacklist
Global: "Only allowed/** with Note:CRUD is accessible"
Key: "Everything EXCEPT allowed/** Note:U is accessible for this key"
Enforcement:
1. Global whitelist resolves first → only allowed/** Note:CRUD is in scope
2. Key blacklist removes Note:U from that scope
Result: Key can do Note:CRD in allowed/**. Update is blocked by the key's blacklist.
Combo 3: Global=Blacklist, Key=Whitelist
Global: "Everything EXCEPT secret/** is accessible"
Key: "Only allowed/** with Note:CR is accessible for this key"
Enforcement:
1. Global blacklist resolves first → everything except secret/** is in scope
2. Key whitelist narrows to only allowed/** Note:CR
Result: Key can Create and Read notes in allowed/** (which is not in global's blacklist, so it's fine).
Combo 4: Global=Blacklist, Key=Blacklist
Global: "Everything EXCEPT secret/** is accessible"
Key: "Everything EXCEPT drafts/** Note:D is accessible for this key"
Enforcement:
1. Global blacklist removes secret/** entirely
2. Key blacklist additionally removes Note:D from drafts/**
Result: Key can access everything except secret/** (global block) and cannot delete notes in drafts/** (key block).
Enforcement algorithm (pseudocode):
function isAllowed(request: { path, dataType, operation, keyId }): boolean { // Step 1: Resolve global scope const globalPerms = resolveScope(global.security, request.path); if (!globalPerms) return false; // path not in global scope at all // Step 2: Check global permission for this operation if (!globalPerms[request.dataType][request.operation]) return false; // Step 3: Resolve key scope (independent listMode) const keyPerms = resolveScope(key, request.path); if (!keyPerms) return false; // path not in key's scope // Step 4: Check key permission (constrained by global) return keyPerms[request.dataType][request.operation]; } function resolveScope(scope: { listMode, paths }, requestPath): Perms | null { const match = scope.paths.find(p => matchPath(requestPath, p.path)); if (scope.listMode === 'whitelist') { // Whitelist: only matched paths are allowed return match ? match.permissions : null; } else { // Blacklist: matched paths have their perms BLOCKED if (!match) return ALL_PERMISSIONS; // not blacklisted → full access return invertPermissions(match.permissions); // flip: set = blocked } }
Key insight: resolveScope() handles listMode interpretation per layer.
The outer function just intersects global result with key result — two independent checks, always AND.
Global can never be bypassed by key configuration. Key can only further restrict.
